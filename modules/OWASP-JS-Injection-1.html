SQL Injection: Bypass Login

Notion Export · 24 April 2026 #SQLi #AuthBypass Manual injection · browser only OWASP Top 10: A03:2021

Ringkasan eksekutif: Berdasarkan eksperimen pada aplikasi vulnerable (Juice Shop / platform serupa), berhasil melakukan login bypass untuk akun admin, Jim, dan Bender hanya menggunakan form login browser + teknik SQL injection. Tidak diperlukan tool otomatis — hanya karakter ' , OR 1=1 dan -- (komentar SQL). Berikut rincian metode & payload yang digunakan.

1. Admin — Bypass total & privilege escalation

Target pertama adalah akun administrator. Melalui reconnaissance pasif (membaca komentar/review pada halaman produk) ditemukan email admin: admin@juice-sh.op. Dengan celah injeksi SQL pada parameter email, login dapat dipalsukan.

PAYLOAD #1 (specific email) 
admin@juice-sh.op' --
Hasil: berhasil login sebagai admin tanpa password. Query SQL diabaikan dari tanda -- ke belakang.
PAYLOAD #2 (blind universal / OR 1=1) 
penghoetank' OR 1=1 --
Hasil: kondisi OR 1=1 selalu bernilai TRUE, database mengembalikan user pertama (biasanya admin) dan login sukses. Apapun string sebelum ' tidak penting.

📖 Penjelasan teknis (mekanisme injeksi)

Aplikasi rentan karena menyusun query SQL secara dinamis tanpa prepared statement. Kira-kira bentuk asli query:

SELECT * FROM users WHERE email = '$email' AND password = '$password'

Dengan injeksi payload admin@juice-sh.op' -- , ekspresi berubah menjadi:

SELECT * FROM users WHERE email = 'admin@juice-sh.op' -- ' AND password = 'anything'

Karakter -- (double dash + spasi) mengomentari sisa query, sehingga password diabaikan. Untuk payload penghoetank' OR 1=1 --, kondisi menjadi:

SELECT * FROM users WHERE email = 'penghoetank' OR 1=1 -- ' AND password = '...'

Karena 1=1 bernilai benar, query mengambil semua baris, lalu aplikasi biasanya memakai data baris pertama → admin.

2. Akun lain: Jim & Bender (manual bypass)

Prinsip serupa digunakan untuk pengguna Jim dan Bender. Langkah pertama: menemukan alamat email mereka yang terdaftar di sistem (biasanya melalui halaman profil publik, komentar, atau daftar user yang terekspos). Setelah mendapatkan email spesifik, injeksi dengan ' -- langsung berhasil.

🎯 Login sebagai Jim

jim@juice-sh.op' --
email ditemukan bypass password

Cara: masukkan email tersebut ke field login, lalu tambahkan kutip tunggal dan spasi + --. Biarkan password kosong / sembarang. Login sukses mengakses akun Jim.

🤖 Login sebagai Bender

bender@juice-sh.op' --
email diketahui bypass auth

Teknik identik: bender@juice-sh.op' -- akan mengabaikan verifikasi password dan langsung login sebagai Bender.

📌 Mengapa teknik ini bekerja?

  • Backend tidak melakukan escaping/parameterisasi pada input email.
  • Simbol ' menutup string SQL, lalu -- menjadikan sisa query sebagai komentar (termasuk klausa password).
  • Database akan mencocokkan email yang valid, dan tidak peduli dengan nilai password karena sudah dikomentari.
  • Ini merupakan classic tautological injection pada mekanisme login.

Insight & Catatan Pribadi

Hasil observasi lapangan:
• Metode OR 1=1 sangat efektif untuk blind attack ketika kita tidak tahu email admin sekalipun.
• Untuk user Jim & Bender, enumerasi email sangat kritis. Sumber: daftar komentar publik, fitur "forgot password" yang leak info, atau galeri user.
• Seluruh percobaan dilakukan hanya menggunakan browser (developer tools tidak wajib), cukup ubah input field.
• Penting: setelah sukses login, kita bisa mengeksplorasi data sensitif, mengubah hak akses (post-exploitation).
• Pada versi aplikasi yang sudah di-patch, serangan gagal — menekankan pentingnya sanitasi input serta parameterized queries.

Refleksi keamanan (mitigasi)

  • Prepared statements / ORM – tidak ada lagi concatenation query string.
  • Validasi input ketat – filter karakter seperti ', --, dan OR 1=1 di sisi server.
  • WAF (Web Application Firewall) – dapat mendeteksi pola SQLi sederhana.
  • Least privilege database – akun aplikasi hanya memiliki akses terbatas.
